Burst Buffer Research

Burst Buffer?

버스트 버퍼는 무엇인가?
이름을 통해 우리는 대략적으로 버스트 버퍼라는 것이 “burst한 무언가에 대한 버퍼”라는 것을 알 수 있다.
버스트 버퍼는 고성능 컴퓨팅 환경에서 bursty한 I/O 처리를 위한 스토리지 서브시스템이다.

Emerging the Burst Buffer

그렇다면 버스트 버퍼가 등장한 배경은 무엇일가?
컴퓨팅 파워 증가로 인한 데이터 양의 증가가 그 배경이 된다.
고성능 컴퓨팅 시스템의 컴퓨팅 파워가 증가함에 따라 computing phase와 I/O phase로 이루어진 High Performance Computing(HPC) 응용의 computing phase가 단축되면서 시간당 생성되는 데이터의 양이 증가하게 되었다. 그에 따라 기존 시스템에서의 I/O 요구사항이 높아지게 되었다.

하지만, HDD 기반으로 구성되어 있는 대부분의 Parallel File System(PFS)는 HPC 환경에서 요구하는 I/O 요구사항을 충족시켜주지 못하고 있는 상황이다. 간단한 방법으로 PFS에 HDD를 추가하여 I/O 대역폭을 늘리는 방법이 있지만, 이는 비용대비 매우 효율적이지 않아 해결책으로서 사용되지 않고 있다.

이러한 배경속에서 “버스트 버퍼“ 기술이 등장하였으며, 고성능 스토리지 디바이스로 구성된 버스트 버퍼는 HPC 환경에서 요구하는 충분한 I/O 성능을 제공한다.
버스트 버퍼는 컴퓨팅 노드와 PFS 사이에 위치하여 컴퓨팅 노드에서 발생하는 응용들의 I/O를 고속으로 처리해주며, PFS로의 직접인 접근 제어 및 데이터 필터링을 통해 PFS의 congestion을 줄여주는 I/O middleware의 역할을 수행한다.

Burst Buffer Architectures

버스트 버퍼 구조는 대표적으로 node-local BB 그리고 remote shared BB로 구성될 수 있다. 이때 remote shared BB는 위치에 따라 I/O node BB 그리고 dedicated node BB로 나뉠 수 있다.
각각의 버스트 버퍼 구조에 따른 장단점이 있으며, 이것은 시스템의 목적 및 요구사항 그리고 비용에 따라 선호하는 버스트 버퍼 구조가 달라진다.

Node-local BB

해당 구조는 BB가 컴퓨팅 노드에 위치하며 로컬 노드의 스토리지 스택을 사용한다.
또한 PFS로의 데이터 flushing을 위해서는 별도의 전용 스레드 또는 데몬을 통한 관리가 필요하며 노드간의 데이터 공유가 어렵다는 특징을 가진다.

  • Advantages

    • 스토리지 디바이스를 단일 잡이 독점적으로 사용할 수 있어 일관되고 예측가능한 성능 제공
    • 노드 수 증가에 따른 BB 성능 선형적 증가
    • 네트워크 트래픽 발생하지 않음

  • Disadvantages

    • 기본적으로 공유 네임스페이스를 지원하지 않아 HPC 응용의 주요 I/O pattern 중 하나인 N-1 패턴 및 데이터 공유를 지원하지 않음
    • Node failure에 따른 Data resiliency를 지원하지 않음

Node-local BB 구조는 상당히 이점이 많은 구조이며, 앞으로 PFS가 All-Flash로 대체될 때 그나마 살아남을 수 있는 구조라 생각한다.
Node-local BB의 단점은 컴퓨팅 노드의 PFS와 같은 네트워크 기반의 파일시스템을 사용하여 N-1 지원 및 데이터 공유 기능을 지원함으로써 극복될 수는 있다. 하지만 그에 따른 컴퓨팅 노드 및 네트워크 리소스 사용 그리고 복잡도 증가와 같은 tradeoff가 발생한다.

Node-local BB에 적합한 파일시스템으로 LANL에서 개발한 PLFS(Parallel Log-structure File System)라는 병렬 응용을 위한 체크포인트용 파일시스템이 있다. PLFS는 N-1 I/O 패턴으로 만들어지는 single shared file 형태의 체크포인트 파일을 N-N 형태로 변환시킴으로써 랜덤 write을 연속적인 write 형태로 바꾼다. 이를 통해 백엔드 파일시스템의 성능을 보장 해준다. PLFS 2.0부터는 각 프로세서에서 생성되는 데이터를 임의의 위치에 저장하는 옵션을 제공하여 PLFS를 통한 Node-local BB를 구성할 수 있다. 따라서 PLFS를 활용함으로써 Node-local BB의 한계였던 N-1 패턴 효과적으로 지원할 수 있게 되었다.
하지만, PLFS는 구현 특성상 read 성능이 매우 느리며, 노드간의 데이터 공유를 지원하지 않는 한계를 가지고 있다.

Remote shared BB

해당 구조는 BB가 컴퓨팅 노드가 아닌 I/O 노드 또는 BB 전용의 노드에 위치하는 구조이다.
대부분 동일한 네임스페이스를 제공하는 fast resource pool 형태로 구현되어지며, 컴퓨팅 노드가 직접 BB 네임스페이스를 직접 마운트하는 형태로 사용되어진다.

  • Advantages

    • N-1 패턴 지원
    • 데이터 공유
    • BB 관리의 편의성 및 응용과의 리소스 사용 분리

  • Disadvantages

    • BB 공유에 따른 데이터 mixed으로 인한 PFS 성능 영향
    • 네트워크 대역폭, 노드 수, 슬롯 수에 따른 BB의 I/O 대역폭의 제한
      • I/O node BB
        • PFS와의 네트워크 공유로 인한 PFS의 성능 영향
      • Dedicated node BB
        • 전용 노드 구축으로 인한 비용문제

Remote shared BB 구조는 가장 일반적으로 사용되는 구조로 많은 슈퍼컴퓨터 시스템에서 채택되어 사용되고 있다.
비용의 문제에서 어느정도 자유롭다면 전용의 버스트 버퍼 노드를 충분히 구축하여 운영하는 것이 가장좋은 선택이라고 할 수 있다.
하지만 해당 구조는 비용대비 효율이 좋은 모델은 아니지만, 현재는 해당 구조를 기반으로 많은 연구가 진행되고 있는 상황이다.

Burst Buffe Use Cases

  • Checkpoint/Restart를 위한 checkpoint 파일 저장소
  • Stage in/out 제공
  • Write-through Cache 역할
  • Data sharing 제공
  • Global write 정보를 활용한 reshaping writes
